# Realtime 1.5 Upgrade Design

## Goal

Upgrade amplifier-voice from `gpt-realtime` to `gpt-realtime-1.5`, introducing a server-side sideband control plane, async function calling with a two-tool model, and retention ratio truncation for long-running sessions.

## Background

The current architecture routes all control events through the browser: the browser receives tool call events from OpenAI via the WebRTC data channel, forwards them to the voice server over HTTP, and injects results back into the OpenAI session. This works but places the browser in the critical path for all intelligence, limits our ability to do async work, and means the session dies if the browser hiccups.

The `gpt-realtime-1.5` model and the GA Realtime API introduce three capabilities that resolve these issues:

1. **Sideband WebSocket connections** -- a server can connect to the same OpenAI session the browser is using, giving us a full server-side control plane.
2. **Async function calling** -- tool results can arrive at any time, enabling fire-and-forget dispatch of long-running agent work.
3. **Retention ratio truncation** -- automatic context pruning that prevents sessions from hitting the 32K token wall.

## Approach

All three features are implemented together because they are architecturally interdependent. The sideband is the prerequisite for async function calling (the server needs its own connection to inject late-arriving results). Retention ratio truncation is configured via the sideband. The model swap to `gpt-realtime-1.5` is required for the improved behavior that makes the two-tool model work cleanly.

The browser becomes a pure audio transport. All tool routing, context management, and instruction updates move to the server.

## Section 2: Sideband Connection -- Server-Side Control Plane

### Architecture Shift

The browser becomes just an audio transport. All intelligence lives on the server, connected to the same OpenAI session via a sideband WebSocket.

**Current architecture:**

```
Browser <-> OpenAI (WebRTC audio + data channel events)
Browser  -> Server (HTTP POST /execute/{tool_name}) -> Amplifier
Browser  <- Server (HTTP response with result)
Browser  -> OpenAI (conversation.item.create + response.create)
```

**New architecture:**

```
Browser <-> OpenAI (WebRTC audio ONLY)
Server  <-> OpenAI (sideband WebSocket -- ALL control events)
Server  <-> Amplifier (direct Python, no HTTP)
Server   -> Browser (SSE -- UI status updates only)
```

### New Module: `voice-server/voice_server/sideband.py`

Approximately 200 lines, modeled after the `os-idea` VoiceSideband pattern.

### Connection Flow

| Step | Action | Details |
|------|--------|---------|
| 1 | `POST /session` (existing) | Server calls `/v1/realtime/client_secrets`. Change: also extract and store the `session_id` from the response (currently discarded). |
| 2 | SDP exchange via `POST /sdp` (existing) | Browser creates WebRTC connection. Change: extract `call_id` from SDP response `Location` header and return it to the browser. |
| 3 | `POST /voice/sideband` (new) | Browser sends `{call_id, ephemeral_key}` after WebRTC is established. Server opens `wss://api.openai.com/v1/realtime?call_id={call_id}` with the ephemeral key as Bearer auth. |
| 4 | `_listen_loop()` starts | An `asyncio.create_task` that processes all events from the sideband WebSocket. |

### Sideband Event Handling

**Listens for:**

| Event | Action |
|-------|--------|
| `response.function_call_arguments.done` | Routes to `AmplifierBridge.execute_tool()` |
| `input_audio_buffer.speech_started` / transcript events | Forwarded to SSE `/events` for UI |
| `error` | Logged, not fatal |

**Sends:**

| Message | Purpose |
|---------|---------|
| `conversation.item.create` with `function_call_output` | Tool results |
| `response.create` | Trigger model to speak after tool completion |
| `session.update` | Truncation config, instruction updates |

### Lifecycle

The sideband lives as long as the WebRTC session. Destroyed on `POST /voice/end` (new endpoint) or on WebSocket disconnect. A registry on the server maps `call_id` to `VoiceSideband` instance.

### Browser Data Channel

After initial setup, the browser data channel becomes read-only. The browser sends `session.update` at connect time (VAD config, noise reduction, transcription), then goes passive. Mic mute/unmute is a pure browser concern -- mute the audio track, no model involvement.

### Authentication

Uses the ephemeral key (not the master API key). Scoped to that session and short-lived.

## Section 3: Async Function Calling -- delegate + dispatch

### Two-Tool Model

The current single tool is split into two with different execution semantics.

### `delegate` -- Synchronous

For quick operations that complete in roughly 1-5 seconds.

- Examples: "What time is it?", "Read that file", quick lookups.
- Sideband receives `response.function_call_arguments.done`, executes via `AmplifierBridge.execute_tool()`, injects result, sends `response.create`.
- Model waits for the result (brief pause, acceptable).
- This is the existing tool from amplifier-foundation with full agent list and descriptions. No changes to its schema.

### `dispatch` -- Fire-and-Forget

For heavy agent work that takes 10 seconds to 5 minutes.

- Examples: "Explore the codebase", "Implement that feature", "Research this topic."
- Sideband receives the tool call, immediately returns a synthetic result: `"Dispatched to {agent}. You'll be notified when complete."`
- Model says "I'm working on that" and continues conversing.
- Server spawns `asyncio.create_task(run_agent_job(...))` for the real Amplifier agent work.
- When the agent finishes, the server calls `sideband.inject_result(call_id, result)`.

### `dispatch` Tool Definition

`dispatch` is a new lightweight tool module. It does not duplicate the agent list from `delegate`. Its tool description:

> Fire-and-forget async delegation for complex tasks that take time. Accepts the same agents as the delegate tool. Returns immediately -- the user will be notified when the task completes.

**Parameters:**

| Parameter | Type | Description |
|-----------|------|-------------|
| `agent` | string | Which specialist agent to use. Accepts any agent available in the delegate tool. |
| `instruction` | string | What the agent should do. |
| `context_depth` | integer (optional) | How much conversation context to include. |

### Result Injection for `dispatch`

When a dispatched task completes:

1. Server sends `conversation.item.create` with `type: "function_call_output"`, the original `call_id`, and the result text.
2. Server sends `response.create` with per-response instructions:

```json
{
  "type": "response.create",
  "response": {
    "instructions": "A previously dispatched task just completed. Summarize the result concisely for the user."
  }
}
```

3. The model sees the tool result appear in its conversation context and speaks it naturally. From the model's perspective, it is identical to a `delegate` result arriving -- it just happened to arrive later, possibly mid-conversation about something else.

**Edge case -- result arrives while user is speaking:** The sideband injects the result, but the model will not interrupt the user. It waits for the next turn boundary to narrate the result.

### Execution Lock

A single `asyncio.Lock` serializes `delegate` calls (they share the Amplifier session). `dispatch` calls acquire the lock only during the actual `execute()`, releasing between agent turns so `delegate` calls can interleave.

### Client-Side Tool Changes

| Tool | Change |
|------|--------|
| `pause_replies` | Removed. Mic mute is a pure browser audio track operation. |
| `resume_replies` | Removed. Same reason. |
| `cancel_current_task` | Stays, but moves to server-side handling through the sideband. |

## Section 4: Retention Ratio Truncation

### Configuration

The sideband sends a `session.update` immediately after connecting, before any tool call events:

```json
{
  "type": "session.update",
  "session": {
    "truncation": {
      "type": "retention_ratio",
      "retention_ratio": 0.8
    }
  }
}
```

`retention_ratio: 0.8` means: when context fills up, drop the oldest 20% in one chunk. The chunked approach is deliberate -- dropping 20% at once is more cache-friendly than trimming one message at a time, because prompt caching works on stable prefixes. The more prefix that stays identical between turns, the more cache hits.

### Config Surface

The `0.8` value lives in `config.py` as a new field on `RealtimeSettings`, configurable via `RETENTION_RATIO` env var. Default `0.8`, valid range `0.0` to `1.0`.

### Impact on Session Duration

Today the app warns at 55 minutes (`sessionWarningMs` in `useVoiceChat.ts`). With retention ratio truncation, sessions can theoretically run indefinitely since old context gets pruned. The warning stays as a safety net but should fire much less often.

## Section 5: Future Work -- Planned

These are documented with enough detail to pick up in a subsequent round. No code changes now.

### Image/Vision Input

The GA Realtime API supports `input_image` content type alongside audio. Users could share screenshots, error messages, code snippets, or diagrams while talking.

**Implementation path:** New `POST /voice/image` endpoint that accepts an image. The sideband injects it as a `conversation.item.create` with `input_image` content type. The client needs a "share screen" or "upload image" UI affordance.

**Pricing:** $5.00/1M image input tokens, $0.50 cached.

### Configurable Idle Timeouts

The GA API supports `idle_timeout_ms` on VAD turn detection settings, enabling an automatic "Are you still there?" after configurable silence.

**Implementation path:** New config field `idle_timeout_ms` on `RealtimeSettings`, sent via sideband `session.update`. Default: disabled (current behavior). When enabled, OpenAI emits `input_audio_buffer.timeout_triggered`.

Nice-to-have: voice-controllable ("set idle timeout to 30 seconds").

### Audio Output Speed Control

The GA API supports `audio.output.speed` in session config. Users could say "speak faster" or "speak slower" during conversation.

**Implementation path:** Expose as a voice-controllable tool or UI slider. The sideband sends `session.update` with the new speed value.

**Note:** Voice cannot be changed mid-session after the first audio response, but speed can.

## Data Flow Summary

### Synchronous Tool Call (delegate)

```
1. User speaks                    -> OpenAI processes audio
2. Model decides to call delegate -> sideband receives function_call_arguments.done
3. Sideband routes to AmplifierBridge.execute_tool()
4. Amplifier agent executes       -> returns result (~1-5s)
5. Sideband sends conversation.item.create (function_call_output)
6. Sideband sends response.create
7. Model speaks the result
```

### Asynchronous Tool Call (dispatch)

```
1.  User speaks                        -> OpenAI processes audio
2.  Model decides to call dispatch     -> sideband receives function_call_arguments.done
3.  Sideband returns synthetic ack     -> "Dispatched to {agent}..."
4.  Sideband sends response.create
5.  Model says "Working on that"       -> conversation continues normally
6.  Server spawns asyncio.create_task(run_agent_job(...))
7.  ... minutes pass, user may discuss other topics ...
8.  Agent finishes                     -> server calls sideband.inject_result()
9.  Sideband sends conversation.item.create (function_call_output)
10. Sideband sends response.create with "task completed" instructions
11. Model narrates result at next natural opportunity
```

## Error Handling

| Scenario | Handling |
|----------|----------|
| Sideband WebSocket drops | Log error, attempt reconnect with same `call_id`. If reconnect fails, session degrades (no tool calls). |
| `delegate` tool call fails | Return error message as function_call_output. Model narrates the error to the user. |
| `dispatch` task fails | Inject error as function_call_output with the original `call_id`. Model reports the failure. |
| Ephemeral key expires | Sideband connection closes. New key needed via `/session`. |
| Amplifier bridge unreachable | Tool calls return error results. Model informs user that backend services are unavailable. |

## Testing Strategy

| Layer | Approach |
|-------|----------|
| Sideband connection | Unit test `sideband.py` with a mock WebSocket. Verify event routing, result injection, and lifecycle management. |
| delegate execution | Integration test: send a `response.function_call_arguments.done` event through the sideband, verify the result arrives back as `function_call_output`. |
| dispatch execution | Integration test: verify immediate synthetic ack, verify background task runs, verify late result injection with correct `call_id`. |
| Execution lock | Unit test: verify `delegate` calls serialize, `dispatch` calls release lock between turns. |
| Retention ratio | Verify `session.update` is sent on sideband connect with correct truncation config. |
| End-to-end | Manual test: voice conversation exercising both `delegate` and `dispatch` paths, verifying the model narrates results correctly. |
| Prompt changes | Manual test: verify the model no longer over-narrates tool call details, and routine completions get brief acknowledgments. |

## Open Questions

- **Reconnect strategy for sideband drops:** Should we attempt automatic reconnect, or treat a sideband drop as a session-ending event that requires the user to reconnect?
- **Dispatch concurrency limit:** Should there be a cap on simultaneous `dispatch` tasks per session? The execution lock serializes access to Amplifier, but multiple tasks could queue up.
- **Context depth default for dispatch:** What is the right default for `context_depth` when the model does not specify it? Too little context and the agent lacks information; too much and we pay for tokens that are not relevant.
